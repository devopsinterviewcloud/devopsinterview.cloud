# Security Audit — devopsinterview.cloud

Read-only application-security audit of the Next.js (App Router) storefront. Stack: Vercel, Prisma/Postgres, Supabase Storage (private bucket + signed URLs), Resend, Razorpay (INR) + PayPal (USD). No files were modified.

Scope reviewed: `middleware.ts`, `next.config.js`, `vercel.json`, the two webhook routes and their verifiers, the download route + token library, checkout route, server-authoritative pricing, storage/secret handling, rate limiter(s), security logger, validation helpers, Prisma schema and query usage, the dynamic `[slug]` / `[token]` routes.

## Summary by severity

| Severity | Count |
|----------|-------|
| Critical | 2 |
| High     | 3 |
| Medium   | 4 |
| Low      | 4 |

The webhook signature verification (the most security-critical control) is implemented **correctly**: raw body is read before parsing, HMAC is compared with `crypto.timingSafeEqual`, PayPal defers to PayPal's verify-signature API, and fulfilment is webhook-driven and idempotent. The serious findings below are about controls that are silently non-functional (rate limiting), missing reconciliation (amount/currency), and a fulfilment path that emails downloads even when the PayPal capture failed.

## Findings (most serious first)

| # | Severity | Location | Issue | Concrete fix |
|---|----------|----------|-------|--------------|
| 1 | **Critical** | `middleware.ts:61-74` (with `lib/rate-limiter.ts:34-78`) | **Rate limiting is completely non-functional.** Middleware does `try { await rateLimiter.check(request) } catch { return 429 }`, but `RateLimiter.check()` never throws — it returns `{ success: false, ... }` on limit. The catch block (the only place a 429 is produced) is dead code, so `/api/checkout` and `/api/download` are effectively unlimited. Card-testing on checkout and brute-force/enumeration on download links are unthrottled. | Inspect the return value: `const r = await rateLimiter.check(request); if (!r.success) { logRateLimit(...); return new NextResponse('Too Many Requests', { status: 429, headers: {...} }) }`. Better: switch to `enforceRateLimit` / `checkRateLimitRedis` from `lib/rate-limiter-redis.ts` which is designed for this. Add a test that asserts the Nth request returns 429. |
| 2 | **Critical** | `middleware.ts:2` + `lib/rate-limiter.ts:17-26,80-91` | **In-memory limiter is ineffective on Vercel serverless even once finding #1 is fixed.** State lives in a per-instance object and is reset on every cold start; each concurrent Lambda has its own counter, and `setInterval` is not reliable in the edge/serverless lifecycle. The `clientId` is derived from `x-forwarded-for[0]`, which a client can spoof when not behind Vercel's trusted proxy normalization. The repo already ships the correct fix (`lib/rate-limiter-redis.ts`, Upstash) but middleware does not use it. | Use the Upstash-backed limiter (`lib/rate-limiter-redis.ts`) in middleware and provision `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN`. Note that the Redis limiter currently "fails open" on Redis error (`lib/rate-limiter-redis.ts:97-107`); for payment endpoints prefer fail-closed or at least alert. Derive the client IP from a trusted header only. |
| 3 | **High** | `src/app/api/webhooks/paypal/route.ts:18-23` + `src/lib/payments/paypal.ts:48-56` | **Download is emailed even when the PayPal capture fails.** On `CHECKOUT.ORDER.APPROVED`, `capturePayPalOrder()` is called inside `try { ... } catch (e) { console.error(...) }` and then `fulfillByGatewayOrderId()` runs **unconditionally**. If capture throws (insufficient funds, already-voided, API error) the order is still marked `SUCCEEDED`/`COMPLETED` and the customer is emailed a working download link — goods delivered without payment. | Only fulfil on a confirmed capture. Either fulfil exclusively on the `PAYMENT.CAPTURE.COMPLETED` webhook (it already carries the order id), or check the capture response status is `COMPLETED` before calling `fulfillByGatewayOrderId`, and do not swallow the capture error. |
| 4 | **High** | `src/app/api/webhooks/razorpay/route.ts:19-25`, `src/app/api/webhooks/paypal/route.ts:24-29`, `src/lib/fulfillment.ts:12-30` | **No amount / currency reconciliation at fulfilment.** Fulfilment keys only on `gatewayOrderId` + event type; it never compares the gateway-reported captured amount/currency against the `Order.total`/`currency` stored at checkout. Signature verification makes forgery hard, but there is no defence-in-depth against a mismatched/partial capture or a misconfigured webhook delivering an unrelated event. | In `fulfillByGatewayOrderId`, pass the captured amount + currency from the webhook payload (`payload.payment.entity.amount`/`currency` for Razorpay; the capture `amount` for PayPal) and assert they equal the stored order before marking it paid. Reject and log on mismatch. |
| 5 | **High** | `src/app/api/download/[token]/route.ts:9-46`, `src/lib/download-token.ts:19` | **Download tokens are bearer tokens, not single-use, valid 3 days.** Any holder of the link (forwarded email, mail-provider link scanning, browser history, leaked `Referer` when the multi-file HTML page links to `https:` Supabase URLs) can re-download repeatedly for 3 days. There is no per-token download count, no IP/session binding, and no revocation. Entitlement check itself (order exists + slug matches + `SUCCEEDED`) is correct. | Reduce TTL (e.g. 24h), and record/limit downloads (the `Download` table already exists) — cap N downloads per token or mark consumed after first success. Consider binding the token to the purchasing email and/or returning signed URLs only to an authenticated session. Set `Referrer-Policy: no-referrer` on the download response so signed URLs don't leak via `Referer`. |
| 6 | **Medium** | `src/app/api/health/route.ts:43-46,71-75` | **Health endpoint leaks raw database error messages** to anonymous callers (`dbError = error.message` returned in the JSON). Prisma/Postgres errors can disclose host, database name, driver details, and connectivity topology. The route is unauthenticated and `force-dynamic`. | Return a generic `"down"` status without `error` to unauthenticated callers; log the detail server-side only. Optionally gate the verbose body behind an internal token or restrict to Vercel cron/monitoring. |
| 7 | **Medium** | `middleware.ts:16-18` | **CSP allows `'unsafe-inline'` in `script-src` in production.** This materially weakens XSS protection (any injected inline `<script>` executes). It is present to support Razorpay/PayPal/GTM inline snippets. `img-src 'self' data: https: blob:` is also very broad. | Move to nonce- or hash-based inline scripts (Next.js supports a CSP nonce via middleware) and drop `'unsafe-inline'`. Tighten `img-src` to the specific hosts already enumerated in `next.config.js` `images.remotePatterns` instead of all `https:`. |
| 8 | **Medium** | `middleware.ts:6-30` vs `next.config.js:26-49` | **Security headers are defined in two places and have diverged.** Middleware sets the full set incl. CSP with `Strict-Transport-Security: max-age=31536000`; `next.config.js` sets an overlapping set with a *different* HSTS `max-age=63072000` and **no CSP**. For static asset paths excluded from the middleware matcher (`middleware.ts:99-108`), only the weaker `next.config.js` headers (no CSP) apply, and the conflicting HSTS values are a maintenance hazard. | Pick one source of truth (middleware is the natural choice since CSP must be dynamic). Remove the duplicated header block from `next.config.js` or keep only headers middleware cannot set, and reconcile the HSTS `max-age`. |
| 9 | **Medium** | `lib/rate-limiter-redis.ts:97-107` | **Rate limiter fails open on Redis errors.** If Upstash is unreachable, every request is allowed (`success: true`). For `/api/checkout` (payment fraud / card testing) this removes the control exactly when it may be under load/attack. | For payment-sensitive endpoints, fail closed (return 429/503) or implement a short in-process fallback budget. At minimum emit a high-severity security log on Redis failure so it is noticed. |
| 10 | **Low** | `lib/` vs `src/lib/` (e.g. `rate-limiter.ts`, `security-logger.ts`, `email.ts`) | **Duplicate divergent copies of security-relevant modules.** `middleware.ts` imports from top-level `lib/` while routes import from `src/lib/`. The two trees can (and the rate-limiter does) drift, so a fix applied to one is silently not applied to the path actually executed. | Consolidate to a single module per concern and have both middleware and routes import it. Delete the dead copy. |
| 11 | **Low** | `src/lib/security-logger.ts:12-41`, `src/lib/rate-limiter.ts:13-26` | **Security logging and counters are in-memory only on serverless.** `SecurityLogger` keeps the last 1000 events in a per-instance array and `sendToMonitoringService` is a no-op stub; on Vercel these events are lost on every cold start and never aggregated, so rate-limit/suspicious-activity logs are effectively unusable for detection. | Ship security events to the existing Sentry integration (already configured) or an external sink; do not rely on in-process arrays for audit/detection on serverless. |
| 12 | **Low** | `src/lib/validation.ts:30-37,204-215` | **Dead/weak validation helpers not wired to live routes; in-house XSS regex and origin/CSRF check are unreliable.** The `checkout`/`download` Zod schemas and `secureValidateRequest` CSRF/origin logic in this file are not used by the actual `src/app/api/*` routes (checkout uses its own inline Zod schema). The regex-based "XSS prevention" (`/<script|javascript:|on\w+=/`) and the `origin.includes(host)` substring check are bypassable and give a false sense of protection. | Either delete the unused helpers (to avoid future misuse) or adopt them in the routes with proper allowlist-based origin checks (exact host comparison, not `includes`) and rely on output encoding / framework escaping for XSS rather than input regex. |

## Verified-correct controls (no action needed)

- **Razorpay webhook signature** (`src/lib/payments/razorpay.ts:38-60`): reads the raw text body before parsing (`route.ts:10`), HMAC-SHA256 over the raw body, compared with `crypto.timingSafeEqual` over equal-length buffers, returns false on missing secret/signature. Correct.
- **PayPal webhook verification** (`src/lib/payments/paypal.ts:62-80`): delegates to PayPal's `verify-webhook-signature` API with the transmission headers and `webhook_id`; returns false unless `verification_status === 'SUCCESS'`. Correct (modulo finding #3 about the capture path).
- **Download token integrity** (`src/lib/download-token.ts:27-41`): HMAC-SHA256, constant-time compare, expiry enforced, malformed input rejected. Tokens cannot be forged or have their payload tampered without the secret. (Remaining issue is bearer-token reuse — finding #5.)
- **Server-authoritative pricing** (`src/app/api/checkout/route.ts:28-66`, `src/lib/pricing.ts`): the client sends only `productId` + `currency` + `email`; amounts come exclusively from the server-side catalog `quote()`. The client cannot influence the charged amount. Currency is mapped to a gateway server-side. Good.
- **Secrets handling** (`src/lib/storage.ts:27-35`): Supabase **service-role** key is read only in server modules (`runtime = 'nodejs'` routes), never prefixed `NEXT_PUBLIC_`, never sent to the client. No real secrets are committed — only `.env.example` placeholders are tracked, and `.gitignore` excludes all `.env*` variants. `next.config.js` only exposes `NEXT_PUBLIC_APP_URL` and `NEXT_PUBLIC_GA_MEASUREMENT_ID`.
- **Injection / traversal / SSRF**: no `queryRawUnsafe`/`executeRawUnsafe`; the only raw query is a parameterless `SELECT 1` health check. `[slug]` resolves against a static JSON array (`src/app/ebooks/[slug]/page.tsx`), `[token]` is HMAC-verified before any use, and the slug passed to Supabase is mapped through a fixed `FILE_MAP` allowlist (`src/lib/storage.ts:16-25`) so no attacker-controlled path reaches storage. JSON-LD output escapes `<`. The multi-file download HTML interpolates only server-controlled `FILE_MAP` names, so no stored XSS.
- **Fulfilment idempotency** (`src/lib/fulfillment.ts:18-20`): re-entrant webhooks short-circuit once `SUCCEEDED`; `gatewayOrderId` is `@unique`. Good.
